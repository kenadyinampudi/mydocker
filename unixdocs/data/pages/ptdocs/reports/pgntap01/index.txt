<span class='nav-bar-title'>
Scopes:
</span>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains information about the operating system, name, version, and
architecture of the inspected system.&lt;/p&gt;
' data-toggle='popover' href='#os' title='Operating System'>
<span>OS</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains information on all installed packages installed on the
inspected system.&lt;/p&gt;
' data-toggle='popover' href='#packages' title='Packages'>
<span>PK</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains all patterns or tasks installed on the inspected system. A pattern is a
collection of software packages, similar to the idea of tasks on Debian/Ubuntu-
like systems. The meaning of software patterns depends on the package manager of the
distribution.&lt;/p&gt;
' data-toggle='popover' href='#patterns' title='Patterns'>
<span>PT</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains all information about software repositories configured on the
inspected system. The information about repositories depends on the package
manager of the distribution. Machinery collects the following information
from each configured repository:&lt;/p&gt;

&lt;ul&gt;
  &lt;li&gt;
    &lt;p&gt;The alias name of the repository.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;The repository type, rpm-md and YaST types that are used on SUSE systems.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;The path to the repository. This could be a local path, a remote location,
a device, or a file.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;A boolean flag that indicates if this repository is in use or not.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;A boolean flag that indicates if this repository should update the locally
stored metadata files with metadata files from the origin automatically or
not.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;A boolean flag that indicates if packages which would be installed from
this repository should be checked by their gpg key or not.&lt;/p&gt;
  &lt;/li&gt;
  &lt;li&gt;
    &lt;p&gt;A numeric value for a priority. The priority of a repository is compared
to the priorities of all other activated repositories. Values can
range from 1 (highest) to 99 (lowest, default).&lt;/p&gt;
  &lt;/li&gt;
&lt;/ul&gt;
' data-toggle='popover' href='#repositories' title='Repositories'>
<span>R</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains information about the system users including user and group ids,
login information, such as password hashes and - if available - additional
password properties.&lt;/p&gt;
' data-toggle='popover' href='#users' title='Users'>
<span>U</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Contains information about the system groups such as group attributes and the
list of group members.&lt;/p&gt;
' data-toggle='popover' href='#groups' title='Groups'>
<span>G</span>
</a>
<a class='btn btn-default btn-sm' data-content='&lt;p&gt;Services are applications running in the background doing continuous work
or waiting for requests to do work.
The scope determines which services are configured to be started in which
runlevel. It uses the &lt;code&gt;chkconfig&lt;/code&gt; command to obtain that information.
The xinetd services that are also displayed by &lt;code&gt;chkconfig&lt;/code&gt; are switched
on/off by editing configuration files and are ignored in this context.&lt;/p&gt;
' data-toggle='popover' href='#services' title='Services'>
<span>S</span>
</a>
<a class='btn btn-default btn-sm disabled' data-content='&lt;p&gt;Contains all configuration files which have been changed since they were
installed.
Changed configuration files are all those files which are marked as such in the
package which has installed them. A configuration file change is reported
if its content or its attributes like Linux permission bits or ownership
have changed.&lt;/p&gt;
' data-toggle='popover' href='#changed_config_files' title='Changed Configuration Files'>
<span>CCF</span>
</a>
<a class='btn btn-default btn-sm disabled' data-content='&lt;p&gt;Contains the names and contents of all non-configuration files which have
been changed compared to the files in the package. A file change is reported
if its content or its attributes like Linux permission bits or ownership
have changed.&lt;/p&gt;
' data-toggle='popover' href='#changed_managed_files' title='Changed Managed Files'>
<span>CMF</span>
</a>
<a class='btn btn-default btn-sm disabled' data-content='&lt;p&gt;Contains the names and contents of all files which are not part of any package.
The list of unmanaged files contains only plain files and
directories. Special files like device nodes, named pipes and Unix domain
sockets are ignored. The directories &lt;code&gt;/tmp&lt;/code&gt;,  &lt;code&gt;/var/tmp&lt;/code&gt;, &lt;code&gt;/.snapshots/&lt;/code&gt;,
&lt;code&gt;/var/run&lt;/code&gt; and special mounts like procfs and sysfs are ignored, too.
If a directory is in this list, no file or directory below it belongs to a
package.&lt;/p&gt;

&lt;p&gt;Meta data information of unmanaged files is only available if the files were
extracted during inspection.&lt;/p&gt;

&lt;p&gt;Using the &lt;code&gt;--extract-unmanaged-files&lt;/code&gt; option, the files are transferred from
the system and stored in the system description. Depending on the content of
the inspected system, the amount of data stored may be huge.&lt;/p&gt;
' data-toggle='popover' href='#unmanaged_files' title='Unmanaged Files'>
<span>UF</span>
</a>
</span>
